OTHER terminologies A Uniform Resource Identifier (URI) is a string of characters that unambiguously identifies a particular resource.  The most common form of URI is the Uniform Resource Locator (URL), (web address). URL (Uniform Resource Locator) - An address for a resource on the web, such as https://moatacademy.com/about. A URL consists of a protocol (http://), domain (moatacademy.com), and optional path (/about). A URL describes the location of a specific resource, such as a web page. 

Characteristics of rest api Client-Server : There should be a separation between the server that offers a service, and the client that consumes it.The only information the client application should know about the server is the URI of the requested resource; Similarly, a server application shouldn&apos;t modify the client application other than passing it to the requested data via HTTP Stateless : Each request from a client must contain all the information required by the server to carry out the request; The server cannot store information provided by the client in one request and use it in another request. Cacheable : caching compensates for the statelessness of  REST, The server must indicate to the client if requests can be cached or not through the headers; allowing for faster response times in subsequent requests for the same resource and reduction in server load.

Characteristics of rest api Layered System : Communication between a client and a server should be standardized, the endpoint may decide to execute multiple instructions or involve intermediaries to respond to requests instead of the end server, without the client having to do anything different.     Uniform Interface : The method of communication between a client and a server must be uniform. The specification of the API is a contract between the app and the server. The app needs to know that it can hit the same URI to get a particular piece of data, every-time! It also needs to know that it’ll get the data in the format that it expects, and that format will not change. This is the Contract and the URI.  Code on demand: Servers can provide executable code or scripts for clients to execute in their context. This constraint is the only one that is optional. Moat Academy for Developers

HOW DO CLIENT MAKE A REQUEST?  THE  METHOD s The method is the type of request you send to the server. You can choose from these five types below: GET : This request is used to get a resource from a server. POST: This request is used to create a new resource on a server. PUT: This is used to update /replace  a resource on a se rver PATCH: Same as Put, used to  partially  update a resource on a server DELETE: This request is used to delete a resource from a server HEAD: To retrieve meta data OPTIONS: To get communication options PUT vs POST PUT is  idempotent  in that sending the same request multiple times produces the same result (no side effects after the first call) while POST is  not idempotent  as sending the same request multiple times may result in duplicate resources or different outcomes.

Naming Conventions for YOUR endpoints Consistent naming conventions in REST API endpoints enhance readability, understanding, and troubleshooting; it also makes it possible to acheive scalability. Below are some best practices around naming your endpoints: Separate words with hyphens Use forward slashes Use Nouns Avoid file extensions Avoid special characters The version number is used to isolate clients from changes to the REST API that might be introduced in future releases. CSS Naming Conventions When working on large scale projects, developers might run into bloated code bases due to the presence of legacy or unknown css classes that must not be tampered with. The  BEM  naming convention which stands for  Block ,  Element ,  Modifier  is a naming convention for CSS classes to write clean, readable, and maintainable code, especially in large-scale projects. Block  : a self-contained piece of UI that makes sense on its own, you can re-use it independently across the site. Example: button, form, menu, etc syntax of the class:  .block Element : A part of the block that has no standalone meaning - like an item in a menu or a label in a form. syntax of the class:  .block__element Modifier : A flag on a block or element that changes its appearance or behavior.  syntax of the class:  .block--modifier  or  block__element--modifier

Cookies – How it works A cookie is just a piece of data which the server sets in the browser:  It is stored on a client’s computer in the form of a text file. Its purpose is to remember and track data pertaining to a client’s usage and can be used to create better visitor experience and site statistics. The browser sends the request for a webpage to the server. The server responds to the browser request by sending the requested webpage along with one or more cookies. Relating with the database via ORM Object relational mappings

Interacting with the database via  orm Object-relational mapping (ORM) is a mechanism that makes it possible to address, access and manipulate objects without having to consider how those objects relate to their data sources SQLAlchemy comes with a powerful ORM ( Object Relational Mapper ) which allows us to work with various databases using Object Oriented code instead of writing raw SQL.  In this concept, your database table are treated as Classes while the columns on the table are the classes’ properties. ORM  lets programmers maintain a consistent view of objects over time , even when the source changes. For example if your database changes from MySQL to ORACLE, you don’t have to adjust your code. SQLAlchemy will generate the sql query that matches ORACLE syntax

Interacting with the database via  orm Advantages of Using ORM: It handles: 1-connecting to database server . 2-generating query . 3- sanitizing the parameter if any. 4-fetching the data and serializing it if necessary . 5- fetching related records

In this method, the base template will contain the parts that are not changing  , you will remove the things that change from page to page from the base layout and create markers in the position they are removed from. For inheritance, we will identify the part  that changes across the two pages , create a base layout that has all the parts but create marker(s) to the part that change. The two pages can each inherit the layout and populate the marked areas with their own contents

Template inheritance In the child template, use the  {% extends %}  tag to inherit from the base template. This is declared at the top of the child template We insert the dynamic content inside the  {% block  blockname  %} ... {%  endblock  %}  tags.  Everything else is loaded right from the base template, so we don't have to re-write things that are common to all pages, such as the navigation bar and the footer

Template inheritance When Jinja encounters the extends statement it loads the base template and then replaces the blocks of content in the parent template with the blocks of content of the same name in the child template.  In case, a block of matching name isn’t found in the child template then the block from parent template will be used Data passed to the child template will be accessible from both the parent template as well as the child template.
